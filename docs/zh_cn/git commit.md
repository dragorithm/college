# git commit

提交信息应使用英文书写，以确保一致性和可读性。

如果git文档更新, 提交使用新的规范, 包括git文档本身

## 规则

\[类型\](范围): 消息  
描述

脚注

## 描述

必填

- [类型]
- (范围)
- : 消息

## 类型

- **feat**: 功能
  - add: 引入新功能；可能涉及添加新模块或在现有模块中添加新内容
  - del: 从模块中移除内容或删除整个模块
  - up: 更新现有功能
- **fix**: 缺陷修复
  - security: 修复与安全相关的问题
  - defect: 修复非安全性的缺陷
- **docs**: 更新 ./docs 下的文档与根目录的`README.md`
  - git: 更新与git相关的
  - '': 其它, 不需要子type
- **style**: 格式调整，不影响逻辑
- **refactor**: 代码重写, 无新增功能, 无错误修复
- **perf**: 性能优化
- **revert**: 撤销提交
- **chore**: 其他

## 范围

- **college**: 与学院作业相关的内容
- **docs**: 与 docs 目录或 `README.md` 相关的内容。
    当满足类型是 **docs**时，可忽略
- **git**: 与 Git 相关的内容
- **notes**: 与学习笔记相关的内容
- **vscode**: 与 VS Code 相关的内容

## 核心三层提交结构

- 类型：描述了做了什么。
- 范围：描述了更改的位置。
- 消息：描述做了什么事情。

**注意**: 提交信息应控制在一句话以内，所有文本均应使用小写字母书写，单词之间用空格分隔，且句末不加句号。
    例外情况是专有名词可以使用大写字母，例如 OpenZFS、Toothless 和 Darkstalker。

## 拓展提交结构

- 描述: 仅在需要解释为何进行更改时填写。

*注意*: 在大多数情况下，更改背后的理由应记录在其他文档中，而非提交信息中。
